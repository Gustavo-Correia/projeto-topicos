Single-package Flutter app rooted at `lib/main.dart`, bootstrapping `Hive.initFlutter` and two boxes (`subscriptions_box`, `settings_box`) before launching `MaterialApp`. State is centralized in three `ChangeNotifier`s registered via `MultiProvider`: `SubscriptionProvider` (domain logic for CRUD, filtering, sorting, aggregations like `totalMonthly`/`categoryTotals`), `SettingsProvider` (user profile, onboarding flag, theme selection, display currency), and `CurrencyProvider` (USD→BRL and annual Selic rates fetched from public REST APIs). Providers and services are wired through an explicit `ServiceLocator` (`lib/services/service_locator.dart`) that maps abstract storage interfaces to concrete implementations, keeping dependency injection visible. Providers depend on abstracted storage interfaces (`SubscriptionStorage`, `SettingsStorage`) implemented by `*_storage_service.dart` classes backed by Hive JSON serialization; this lets tests inject fakes. External data is isolated in `services/currency_api_service.dart` (AwesomeAPI USD→BRL + Banco Central SGS Selic), which returns `null` on any failure so the app degrades gracefully offline. Navigation uses named routes centralized in `lib/routes.dart` (`AppRoutes` constants resolved by `onGenerateRoute`), with `MaterialApp` configured via `initialRoute`/`onGenerateRoute`. Domain models live under `models/` (`Subscription`, `AppSettings`) as immutable value objects with `copyWith` + `toMap`/`fromMap` converters. Screens under `screens/` are stateless/stateful entry points that read providers through `context.read`/`Consumer`; the splash screen orchestrates parallel initialization of settings and subscriptions then routes to `HomeScreen` or `OnboardingScreen`. Theming is decoupled into `utils/app_colors.dart` where `AppTheme` maps to a `ThemeData` and `AppColors.of(context)` resolves the active theme from `SettingsProvider`. Reusable UI fragments sit in `widgets/`. Dependency direction: screens → providers → services → models/utils; providers never import screens.
