This repository is a single Flutter project (not a multi-app monorepo) that bundles Dart source, Android/iOS platform shells, tests, and documentation under one pubspec.yaml-driven build. There are no custom Makefiles, shell scripts, Dockerfiles, or CI pipelines committed to the repo; the build system relies entirely on the standard Flutter toolchain plus each platform's native build tools.

What system/approach is used:
- Dart/Flutter SDK for dependency resolution, analysis, testing (flutter test), and cross-platform builds (flutter run, flutter build apk|aab|ios|ipa).
- Android Gradle Plugin with Kotlin DSL via dev.flutter.flutter-gradle-plugin.
- Xcode/CocoaPods for iOS, driven by generated .xcconfig files from flutter create.
- Linting through analysis_options.yaml including package:flutter_lints/flutter.yaml.
- Icon generation via flutter_launcher_icons configured in pubspec.yaml.

Key files and packages:
- pubspec.yaml: package name, version (1.0.0+1), SDK constraint (>=3.11.0 <4.0.0), dependencies, dev dependencies, icon generator config, and Flutter asset declarations.
- analysis_options.yaml: linter rules derived from Flutter lints with project-specific overrides.
- android/build.gradle.kts: root Gradle script centralizing repositories, redirecting all build outputs into shared build/, and registering a clean task.
- android/app/build.gradle.kts: applies com.android.application and dev.flutter.flutter-gradle-plugin, sets compile/target Java/Kotlin 17, pulls compileSdk/minSdk/targetSdk/versionCode/versionName from Flutter, and points flutter { source = "../.." } back at the Dart root.
- android/local.properties: developer-local paths for sdk.dir and flutter.sdk, plus flutter.buildMode, flutter.versionName, flutter.versionCode.
- ios/Flutter/Generated.xcconfig: generated Xcode config pointing at FLUTTER_ROOT, FLUTTER_APPLICATION_PATH, target file, build dir, and build name/number.
- ios/Runner.xcodeproj and ios/Runner/: Xcode project, Swift entry points, storyboards, assets, and Info.plist consumed by xcodebuild.

Architecture and conventions:
- Single-source-of-truth for versioning: pubspec.yaml declares version: 1.0.0+1; Android reads it through flutter.versionCode/flutter.versionName, and iOS mirrors it in Generated.xcconfig (FLUTTER_BUILD_NAME=1.0.0, FLUTTER_BUILD_NUMBER=1). The local local.properties currently pins these values per-developer.
- Shared build output tree: the root build/ directory aggregates artifacts from both Android subprojects and the Flutter engine, keeping generated binaries out of source control (.gitignore excludes /android/app/release).
- No signing configured for release yet: android/app/build.gradle.kts leaves release.signingConfig pointing at debug, with a TODO comment indicating real keystore configuration is pending.
- Assets are referenced directly by path in flutter_launcher_icons but are not listed under flutter.assets: in pubspec.yaml; this works for icon generation but could cause runtime loading failures if code tries to load the image as a bundled asset.

Rules developers should follow:
- Keep pubspec.yaml as the canonical place for app version and dependency updates; do not edit android/local.properties or ios/Flutter/Generated.xcconfig by hand — they are regenerated by Flutter.
- When adding images/fonts, declare them under flutter.assets: in pubspec.yaml so they ship with the app bundle (the current app_icon_master.png is only consumed by the icon generator).
- For Android releases, replace the debug signing stub in android/app/build.gradle.kts with a proper signingConfigs block referencing a keystore before building flutter build apk --release or flutter build appbundle --release.
- Lint changes must pass flutter analyze; the project enforces prefer_const_constructors, prefer_const_literals_to_create_immutables, and avoid_print.
- Tests live under test/ and are executed via flutter test; there is no separate test harness or CI step committed to the repo.
